15//! A Rust implementation of Olm and Megolm
16//!
17//! vodozemac is a Rust reimplementation of
18//! [libolm](https://gitlab.matrix.org/matrix-org/olm), a cryptographic library
19//! used for end-to-end encryption in [Matrix](https://matrix.org). At its core, it
20//! is an implementation of the Olm and Megolm cryptographic ratchets, along
21//! with a high-level API to easily establish cryptographic communication
22//! channels employing those ratchets with other parties. It also implements
23//! some other miscellaneous cryptographic functionality which is useful for
24//! building Matrix clients, such as [SAS][sas].
25//!
26//! [sas]:
27//! <https://spec.matrix.org/v1.2/client-server-api/#short-authentication-string-sas-verification>
28//!
29//! # Olm
30//!
31//! Olm is an implementation of the [Double Ratchet
32//! algorithm](https://whispersystems.org/docs/specifications/doubleratchet/), very
33//! similar to that employed by the Signal Protocol. It allows the establishment
34//! of a 1-to-1 private communication channel, with perfect forward secrecy and
35//! self-healing properties.
36//!
37//! A detailed technical specification can be found at
38//! <https://gitlab.matrix.org/matrix-org/olm/-/blob/master/docs/olm.md>.
39//!
40//! For more information on using vodozemac for Olm, see the [`olm`] module.
41//!
42//! # Megolm
43//!
44//! Megolm is an AES-based single ratchet for group conversations with a large
45//! number of participants, where using Olm would be cost prohibitive because it
46//! would imply encrypting each message individually for each participant.
47//! Megolm sidesteps this by encrypting messages with a symmetric ratchet,
48//! shared once with each participant and then reused for a sequence of messages
49//! before rotating.
50//!
51//! This is a trade-off in which we lose Olm's self-healing properties, because
52//! someone in possession of a Megolm session at a particular state can derive
53//! all future states. However, if the attacker is only able to obtain the
54//! session in a ratcheted state, they cannot use it to decrypt messages
55//! encrypted with an earlier state. This preserves forward secrecy.
56//!
57//! A detailed technical specification can be found at
58//! <https://gitlab.matrix.org/matrix-org/olm/-/blob/master/docs/megolm.md>.
59//!
60//! For more information on using vodozemac for Megolm, see the [`megolm`]
61//! module.
62//!
63//! # Features
64//!
65//! ## Supported
66//!
67//! - [Olm](https://matrix-org.github.io/vodozemac/vodozemac/olm/index.html)
68//! - [Megolm](https://matrix-org.github.io/vodozemac/vodozemac/megolm/index.html)
69//! - [libolm pickle format](#legacy-pickles) (read-only)
70//! - [Modern pickle format](#modern-pickles)
71//! - [SAS (Short Authentication Strings)](https://matrix-org.github.io/vodozemac/vodozemac/sas/index.html)
72//!
73//! ## Unsupported
74//!
75//! - Creating asymmetric [server-side message key
76//!   backups][legacy-message-key-backup], since they are slated to be replaced
77//!   with symmetric backups.
78//!
79//! ## Planned
80//!
81//! - Symmetric [server-side message key backups][symmetric-message-key-backup]
82//! - Importing asymmetric [server-side message key
83//!   backups][legacy-message-key-backup], for compatibility with existing
84//!   backups created by libolm.
85//!
86//! [legacy-message-key-backup]:
87//! <https://spec.matrix.org/v1.2/client-server-api/#server-side-key-backups>
88//!
89//! [symmetric-message-key-backup]:
90//! https://github.com/uhoreg/matrix-doc/blob/symmetric-backups/proposals/3270-symmetric-megolm-backup.md
91//!
92//! # Feature flags
93//!
94//! ## Low-level API
95//!
96//! Feature: `low-level-api` (default: off)
97//!
98//! Vodozemac exposes some lower-level structs and functions that are only
99//! useful in very advanced use cases. These should *not* be needed by the vast
100//! majority of users.
101//!
102//! Extreme care must be taken when using such APIs, as incorrect usage can lead
103//! to broken sessions.
104//!
105//! # Pickling
106//!
107//! vodozemac supports serializing its entire internal state into a form
108//! a "pickle". The state can subsequently be restored from such a pickle
109//! ("unpickled") in order to continue operation. This is used to support some
110//! Matrix features like device dehydration.
111//!
112//! ## Legacy pickles
113//!
114//! The legacy pickle format is a simple binary format used by libolm.
115//! Implemented for interoperability with current clients which are using
116//! libolm. Only *unpickling* is supported.
117//!
118//! ## Modern pickles
119//!
120//! The crate also implements a modern pickling mechanism using
121//! [Serde](https://serde.rs/). The exact serialization format is not mandated
122//! nor specified by this crate, but you can serialize to and deserialize from
123//! any format supported by Serde.
124//!
125//! The following structs support pickling:
126//!
127//! - [`olm::Account`]
128//! - [`olm::Session`]
129//! - [`megolm::GroupSession`]
130//! - [`megolm::InboundGroupSession`]
131//!
132//! For example, the following will print out the JSON representing the
133//! serialized `Account` and will leave no new copies of the account's secrets
134//! in memory:
135//!
136//! ```rust
137//! use anyhow::Result;
138//! use vodozemac::olm::{Account, AccountPickle};
139//!
140//! const PICKLE_KEY: [u8; 32] = [0u8; 32];
141//!
142//! fn main() -> Result<()>{
143//!     let mut account = Account::new();
144//!
145//!     account.generate_one_time_keys(10);
146//!     account.generate_fallback_key();
147//!
148//!     let pickle = account.pickle().encrypt(&PICKLE_KEY);
149//!
150//!     let account2: Account = AccountPickle::from_encrypted(&pickle, &PICKLE_KEY)?.into();
151//!
152//!     assert_eq!(account.identity_keys(), account2.identity_keys());
153//!
154//!     Ok(())
155//! }
156//! ```
157//!
158//! You can unpickle a pickle-able struct directly from its serialized form:
159//!
160//! ```rust
161//! # use anyhow::Result;
162//! # use vodozemac::olm::{Account, AccountPickle};
163//! # use zeroize::Zeroize;
164//! #
165//! # fn main() -> Result<()> {
166//! #   let some_account = Account::new();
167//!     let mut json_str = serde_json::to_string(&some_account.pickle())?;
168//!     // This will produce an account which is identical to `some_account`.
169//!     let account: Account = serde_json::from_str::<AccountPickle>(&json_str)?.into();
170//!
171//!     json_str.zeroize();
172//! #
173//! #    Ok(())
174//! # }
175//! ```
176//!
177//! However, the pickle-able structs do not implement `serde::Serialize`
178//! themselves. If you want to serialize to a format other than JSON, you should
179//! instead call the `.pickle()` method to obtain a special serializable struct.
180//! This struct *does* implement `Serialize` and can therefore be serialized
181//! into any format supported by `serde`. To get back to the original struct
182//! from such as serializable struct, just call `.unpickle()`.
183//!
184//! ```rust
185//! use anyhow::Result;
186//! use vodozemac::olm::Account;
187//!
188//! fn main() -> Result<()> {
189//!     let account = Account::new();
190//!     let account: Account = account.pickle().into();  // this is identity
191//!
192//!     Ok(())
193//! }
194//! ```

218/// Error type describing the various ways Vodozemac pickles can fail to be
219/// decoded.
220#[derive(Debug, thiserror::Error)]
221pub enum PickleError {
222    /// The pickle wasn't valid base64.
223    #[error("The pickle wasn't valid base64: {0}")]
224    Base64(#[from] base64::DecodeError),
225    /// The encrypted pickle could not have been decrypted.
226    #[error("The pickle couldn't be decrypted: {0}")]
227    Decryption(#[from] cipher::DecryptionError),
228    /// The serialized Vodozemac object couldn't be deserialized.
229    #[error("The pickle couldn't be deserialized: {0}")]
230    Serialization(#[from] serde_json::Error),
231}

233/// Error type describing the various ways libolm pickles can fail to be
234/// decoded.
235#[derive(Debug, thiserror::Error)]
236pub enum LibolmPickleError {
237    /// The pickle is missing a valid version.
238    #[error("The pickle doesn't contain a version")]
239    MissingVersion,
240    /// The pickle has a unsupported version.
241    #[error("The pickle uses an unsupported version, expected {0}, got {1}")]
242    Version(u32, u32),
243    /// The pickle wasn't valid base64.
244    #[error("The pickle wasn't valid base64: {0}")]
245    Base64(#[from] Base64DecodeError),
246    /// The pickle could not have been decrypted.
247    #[error("The pickle couldn't be decrypted: {0}")]
248    Decryption(#[from] cipher::DecryptionError),
249    /// The pickle contains an invalid public key.
250    #[error("The pickle contained an invalid ed25519 public key {0}")]
251    PublicKey(#[from] KeyError),
252    /// The pickle does not contain a valid receiving or sending chain. A valid
253    /// Olm session needs to have at least one of them.
254    #[error("The pickle didn't contain a valid Olm session")]
255    InvalidSession,
256    /// The payload of the pickle could not be decoded.
257    #[error(transparent)]
258    Decode(#[from] matrix_pickle::DecodeError),
259    /// The object could not be encoded as a pickle.
260    #[error(transparent)]
261    Encode(#[from] matrix_pickle::EncodeError),
262}

265/// decoded.
266#[derive(Debug, thiserror::Error)]
267pub enum DehydratedDeviceError {
268    /// The pickle is missing a valid version.
269    #[error("The pickle doesn't contain a version")]

272    #[error("The pickle uses an unsupported version, expected {0}, got {1}")]
273    Version(u32, u32),
274    /// Invalid nonce.
275    #[error("The nonce was invalid")]
276    InvalidNonce,
277    /// The pickle wasn't valid base64.
278    #[error("The pickle wasn't valid base64: {0}")]
279    Base64(#[from] Base64DecodeError),
280    /// The pickle could not have been decrypted.
281    #[error("The pickle couldn't be decrypted: {0}")]
282    Decryption(#[from] chacha20poly1305::aead::Error),
283    /// There was an error with the libolm pickle format
284    #[error(transparent)]
285    LibolmPickle(#[from] LibolmPickleError),
286    /// The current account was created by libolm and can't be converted into a
287    /// dehydrated device.
288    #[error("The Account can't be turned into a dehydrated device since it was created by libolm")]
289    InvalidAccount,
290}

292/// Error type describing the different ways message decoding can fail.
293#[derive(Debug, thiserror::Error)]
294pub enum DecodeError {
295    /// The Olm message has an invalid type.
296    #[error("The message has an invalid type, expected 0 or 1, got {0}")]
297    MessageType(usize),
298    /// The message is missing a valid version.
299    #[error("The message didn't contain a version")]
300    MissingVersion,
301    /// The message doesn't have enough data to be correctly decoded.
302    #[error("The message was too short, it didn't contain a valid payload")]
303    MessageTooShort(usize),
304    /// The message has a unsupported version.
305    #[error("The message didn't have a valid version, expected {0}, got {1}")]
306    InvalidVersion(u8, u8),
307    /// An embedded public key couldn't be decoded.
308    #[error("The message contained an invalid public key: {0}")]
309    InvalidKey(#[from] KeyError),
310    /// The embedded message authentication code couldn't be decoded.
311    #[error("The message contained a MAC with an invalid size, expected {0}, got {1}")]
312    InvalidMacLength(usize, usize),
313    /// An embedded signature couldn't be decoded.
314    #[error("The message contained an invalid Signature: {0}")]
315    Signature(#[from] SignatureError),
316    /// The message couldn't be decoded as a valid protocol buffer message.
317    #[error(transparent)]
318    ProtoBufError(#[from] ProtoBufDecodeError),
319    /// The message wasn't valid base64.
320    #[error("The message wasn't valid base64: {0}")]
321    Base64(#[from] Base64DecodeError),
322}

327#[cfg(test)]
328fn corpus_data_path(fuzz_target: &str) -> std::path::PathBuf {
329    let manifest_dir =
330        std::env::var("CARGO_MANIFEST_DIR").expect("Cargo always sets the manifest dir");
331
332    let mut afl_dir = std::path::PathBuf::from(manifest_dir);
333    afl_dir.push("afl");
334    afl_dir.push(fuzz_target);
335    afl_dir.push("in");
336
337    afl_dir
338}
339
340#[cfg(test)]
341fn run_corpus<F>(fuzz_target: &str, method: F)
342where
343    F: FnOnce(&[u8]) + Copy,
344{
345    let dir = corpus_data_path(fuzz_target);
346    let corpus = std::fs::read_dir(dir).expect("Couldn't read the corpus directory");
347
348    for input in corpus {
349        let input = input.expect("Couldn't read the input file");
350        let data = std::fs::read(input.path()).expect("Couldn't read the input file");
351        method(&data)
352    }
353}
